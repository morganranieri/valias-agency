  Prompt: Heya - if I wanted to build a PDF from a rails app, and wanted to do some advance research on the topic, would you have any pointers for me?



  Absolutely. The big decision is **how you want to author the PDF**, because that determines almost everything else.

  For Rails apps today, the main paths are:
  **HTML-to-PDF**, where you render a Rails view and convert it to PDF;
  **Ruby-native PDF generation**, where you build the PDF with a library;
  and **document-pipeline generation**, where content is authored in a markup format and converted to PDF. Wicked PDF is still the classic Rails HTML option and uses `wkhtmltopdf`; Grover is a newer HTML option built around headless Chrome; Prawn is the long-standing pure-Ruby PDF library; and Asciidoctor PDF is strong for structured documents like reports, manuals, and handbooks. ([GitHub][1])

  My strongest pointer is to **choose by document type, not by gem popularity**:

  * **Invoices, receipts, labels, certificates, tightly controlled layouts** → look hard at **Prawn**
  * **“Make this PDF look like our web page / email / admin report”** → start with **Grover** or **Wicked PDF**
  * **Policies, proposals, guides, long reports, multi-page docs with TOC/front matter** → consider **Asciidoctor PDF** ([GitHub][2])

  A good research plan would be:

  **1. Start with the rendering model**
  Ask: do you want to reuse Rails views/CSS, or do you want a PDF-specific layout system?
  Wicked PDF lets you stay in the Rails view world and converts HTML via `wkhtmltopdf`. Grover also fits a Rails HTML workflow and can generate PDF/PNG/JPEG from app pages. Prawn is different: you are not “printing HTML”, you are composing the PDF directly in Ruby. ([GitHub][1])

  **2. Research maintenance and platform risk**
  This matters more than people expect. Wicked PDF itself is established, but it depends on `wkhtmltopdf`, and there has been ongoing community discussion about long-term direction because of that dependency. Grover’s Chrome-based approach is attractive partly because it follows a more modern browser engine. ([GitHub][3])

  **3. Research your exact pain points before choosing**
  Look specifically for:
  page breaks,
  headers/footers,
  custom fonts,
  charts,
  backgrounds,
  Tailwind/modern CSS,
  JS-rendered content,
  performance on large batches,
  and deployment on your hosting stack.
  For example, Grover issues and examples show people using it with Rails 7.x and modern front-end output, while Wicked PDF has a long history of “HTML looks fine in browser, PDF behaves differently” style edge cases. ([GitHub][4])

  **4. Prototype the same document in two approaches**
  This is the fastest way to learn. Build one real sample, such as:

  * an invoice with a table, totals, logo, page numbers
  * a report with headings, charts, page breaks
    Then compare:
    developer effort,
    visual fidelity,
    testability,
    render speed,
    and deployment complexity.
    You will usually know very quickly which path is right.

  My practical recommendation for “advance research” is:

  ### Option A: HTML-first research track

  Research **Grover vs Wicked PDF** if your instinct is “I want Rails templates and CSS.”
  Questions to investigate:

  * Can it render your current CSS stack cleanly?
  * How well does it handle pagination?
  * Do assets, fonts, and absolute/relative URLs behave in production?
  * What happens in Docker / CI / deployment?
  * How do you debug failures?

  Useful starting docs:

  * Wicked PDF README ([GitHub][1])
  * Grover README ([GitHub][2])

  ### Option B: PDF-first research track

  Research **Prawn** if your instinct is “I need reliable document generation, not browser printing.”
  Questions:

  * How much custom layout will I need to hand-code?
  * How painful are tables and repeated headers?
  * How will design changes be maintained over time?
  * Do I need pixel-perfect repeatability more than web-like styling?

  Useful starting doc:

  * Prawn README / docs ([GitHub][5])

  ### Option C: document-publishing research track

  Research **Asciidoctor PDF** if your PDFs are really publications.
  It has native support for structured documents, theming, TOC, title pages, metadata, SVG, and custom fonts. This can be a much better fit than HTML for polished long-form documents. ([Asciidoctor Docs][6])

  A few deeper research questions that separate a quick hack from a solid architecture:

  * **Where is the source of truth?**
    Is the PDF just another presentation of existing Rails data, or is it a formal document with versioning and templates?

  * **Do you need synchronous generation or background jobs?**
    Small PDFs can be request-time. Big reports usually want background generation and storage.

  * **Do you need deterministic tests?**
    HTML-to-PDF can be more brittle than Ruby-native generation. Prawn often gives tighter control, but more custom code.

  * **Will non-developers need to edit templates?**
    If yes, HTML or AsciiDoc may age better than a Ruby layout DSL.

  * **Do you need accessibility, archival, signing, or form filling later?**
    That can affect the choice early.

  My own bias:

  * For a **business app that needs decent-looking PDFs quickly**, I’d research **Grover first**, then **Wicked PDF**
  * For **transactional documents where reliability matters more than web styling**, I’d research **Prawn first**
  * For **manuals, guides, proposals, board papers, or long reports**, I’d seriously consider **Asciidoctor PDF** ([GitHub][2])
